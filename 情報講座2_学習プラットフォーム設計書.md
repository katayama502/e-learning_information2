# 情報講座2 学習プラットフォーム 設計書

高校「情報講座2」向けの学習プラットフォーム設計。
左にスライド／右にPython実行環境を並べ、登録・ログイン → 教材選択 → テスト合格でランクが上がる体験を提供する。

---

## 1. 全体像

```
┌─────────────────────────────────────────────────────────┐
│  学習画面（1レッスン = 1画面）                               │
│  ┌───────────────────────┬───────────────────────────┐ │
│  │  左：スライドビューア       │  右：Pythonエディタ＋実行     │ │
│  │  ・前/次、ページ番号        │  ・コード入力                │ │
│  │  ・教材に同期             │  ・実行ボタン / 出力コンソール  │ │
│  │                        │  ・「課題を提出」              │ │
│  └───────────────────────┴───────────────────────────┘ │
│  下部：進捗バー / 現在ランク / 次ランクまでのXP              │
└─────────────────────────────────────────────────────────┘
```

主要機能は5つ。

1. スライド埋め込み（左）
2. ブラウザ内Python実行（右）
3. 学生の登録・ログイン
4. 教材（ユニット/レッスン）の選択
5. テスト合格でXPが貯まりランクが上がる

---

## 2. 推奨アーキテクチャ

| 層 | 採用技術 | 理由 |
|----|----------|------|
| フロント | Next.js (React) | 画面・APIルート・SSRを一括。Vercelと相性◎ |
| Python実行 | **Pyodide**（WebAssembly） | サーバでコードを動かさない＝**RCE（任意コード実行）リスクを根本排除**。無料。高校Pythonには十分 |
| バックエンド | Next.js API Routes（または FastAPI） | 採点・ランク計算など「信用できる処理」をサーバ側に |
| DB | PostgreSQL（Supabase） | RLSで行レベル権限。無料枠あり |
| 認証 | Supabase Auth | パスワードハッシュ・セッションを安全に肩代わり |
| ホスティング | Vercel + Supabase | 既に接続済みの構成と一致 |

> Pyodideはブラウザ上でPythonを動かすため、**サーバへ生徒のコードが届かず安全**。ただし「採点の信用」だけは別途サーバ側で担保が必要（→ 5章）。

---

## 3. 画面構成（左スライド／右Python）

### 左：スライドビューア
教材スライドの持ち方で実装が変わるため、3案を提示。

| スライド形式 | 埋め込み方法 | 備考 |
|--------------|--------------|------|
| Googleスライド | 公開埋め込みURLを `<iframe>` | 最速。ただし表示制御は弱い |
| PDF | `react-pdf` / PDF.js で自前ビューア | 前/次・現在ページ制御が自由。**推奨** |
| PowerPoint | 画像orPDFへ変換して表示 | 変換ひと手間 |

推奨は **PDF（または画像化）をDBのレッスンに紐づけ、自前ビューアで表示**。レッスンと「右のPython課題」を同期できる。

### 右：Pythonエディタ＋実行
- エディタ：Monaco Editor（VS Code同等）またはCodeMirror
- 実行：Pyodideを **Web Worker** 上で動かす（UIフリーズ防止＋分離）
- 出力：stdout/stderr/例外をコンソール表示
- 「課題を提出」でテストケース判定へ

---

## 4. データモデル（主要テーブル）

| テーブル | 主なカラム | 用途 |
|----------|-----------|------|
| `profiles` | id(=auth.users), display_name, role(student/teacher/admin), xp, rank | 生徒情報・権限・XP |
| `units` | id, title, order | 講座の大単元（例：情報講座2） |
| `materials` | id, unit_id, title, slide_ref, order | 教材＝1レッスン（スライド参照を保持） |
| `questions` | id, material_id, type(choice/code), prompt, choices, correct, points | テスト問題 |
| `code_tests` | id, question_id, input, expected_output | コード課題のテストケース |
| `attempts` | id, user_id, material_id, score, passed, created_at | 受験履歴 |
| `answers` | id, attempt_id, question_id, user_answer, is_correct | 解答明細 |
| `xp_log` | id, user_id, delta, reason, created_at | XP増減の監査ログ |
| `ranks` | name, min_xp | ランク定義（しきい値） |

`role` を持たせることで、生徒／教師（教材・問題の管理）／管理者を分離できる。

## 5. テスト＆採点の設計（整合性が最重要）

問題タイプは2種類。

- **選択式（choice）**：生徒の選択をサーバへ送り、サーバが正誤判定。
- **コード課題（code）**：テストケースで自動判定。

### ⚠ 整合性の注意点
Pyodideは**ブラウザ側**で動くため、点数を改ざんされうる。そこで採点を二層に分ける。

| 区分 | 実行場所 | 採点権限 |
|------|----------|----------|
| 練習・自己学習 | ブラウザ（Pyodide） | 気軽に試せる。点に反映しない |
| **成績に関わるテスト** | サーバ側サンドボックス（Judge0 / Piston など） | **サーバが正本の点数を確定** |

→ ランクに影響する点数は必ずサーバ側で再実行・確定する。これが「ランクの公正さ」を守る肝。

## 6. ランク／XP設計

XPの累積でランクが決まる。情報講座らしいテーマ名の例：

| ランク | 必要XP（累計） |
|--------|----------------|
| ビギナー | 0 |
| ルーキー | 100 |
| ブロンズ | 300 |
| シルバー | 700 |
| ゴールド | 1,500 |
| プラチナ | 3,000 |
| マスター | 6,000 |

XPの付与例：

- 教材スライド読了：+10
- 小テスト合格：+50（満点ボーナス +20）
- コード課題クリア：+30〜80（難易度別）
- 連続学習ストリーク：日数に応じてボーナス

実装ポイント：
- ランクは **累計XPから都度計算**（`ranks` のしきい値を超えたか）。
- XPの増減は必ず `xp_log` に記録（不正・バグの追跡用）。
- 進捗バーで「次のランクまであと◯XP」を表示してモチベ維持。

## 7. 認証・認可

- 登録：メール＋パスワード（必要なら学校ドメイン限定）。Supabase Authが**ハッシュ化・セッション**を担当。
- セッション：httpOnly + Secure + SameSite クッキー。
- 認可：Supabaseの**RLS（行レベルセキュリティ）**で「生徒は自分のデータのみ」参照・更新可。教材/問題の作成は teacher 以上に限定。
- すべてのAPIでサーバ側の権限チェックを実施（フロントの表示制御だけに頼らない）。

---

## 8. セキュリティ設計（OWASP Top 10 を適用）

ユーザー設定のガイダンスを各項目に反映。

| OWASP 項目 | 本設計での対策 |
|------------|----------------|
| アクセス制御の不備 | RLS＋全API権限チェック、role分離、最小権限 |
| 暗号化の失敗 | 全通信HTTPS、PWはSupabaseでハッシュ、秘密情報は環境変数 |
| インジェクション | ORM/パラメータ化クエリ、入力は**zod等でスキーマ検証**、コードはサンドボックス |
| 安全でない設計 | 認証へのレート制限、成績は server-authoritative |
| 設定ミス | 本番はデバッグ無効、CORS適正化、セキュリティヘッダ |
| 脆弱な依存 | `npm audit` 定期実行、依存更新 |
| 認証の失敗 | ログイン回数制限・ロックアウト、強いPWポリシー、必要ならMFA |
| データ完全性 | XP/採点はサーバで確定、`xp_log`で監査 |
| ログ/監視 | 認証イベント・点数変更を記録 |
| SSRF | スライドURL等の外部URLは**許可リスト**で制限 |

機密情報管理：APIキー等は `.env`／Vercel・Supabaseのシークレットに保存し、**コードにハードコードしない・コミットしない**。

> 補足（CSP）：Pyodideは WebAssembly を使うため、Content-Security-Policy に `script-src 'wasm-unsafe-eval'` を許可しないと動かない。厳格化しつつこの例外を入れる。

## 9. 開発フェーズ（ロードマップ）

| フェーズ | 内容 |
|----------|------|
| 1 | 認証＋分割UI（左PDFビューア／右Pyodide）の最小動作 |
| 2 | ユニット/教材の選択・進捗保存 |
| 3 | テスト（選択式＋コード課題）と自動採点 |
| 4 | XP・ランク・ダッシュボード |
| 5 | 教師用管理画面・分析・本番ハードニング |

まずフェーズ1（左右分割＋ブラウザPython＋ログイン）を動くプロトタイプにするのがおすすめ。
